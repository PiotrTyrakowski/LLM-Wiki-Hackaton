Hi — I'm building Cinegraph. It's a video editor agent that doesn't just take a prompt; it watches a human edit, attempts the same edit, scores itself against the human's version, and rewrites its own playbook before trying again. Same agent, smarter every run.

THE IDEA

Video editing is an RL environment in disguise.

  source clips ──►  agent  ──►  edited video
                                  │
                          ▼ compare to ▼
                          human-edited target
                                  │
                          ▼ writes lessons ▼
                          [ Cinegraph wiki ]
                                  │
                          fed back into next attempt

The feedback signal is the actual human edit. The action space is the edit itself.

The trick is: video editing already gives you a reward signal for free — the human's final cut IS the answer key. Compare frame by frame, score, distill what went wrong into the agent's rulebook, retry. No fine-tuning, no labels, no RL infra. Just a wiki that rewrites itself.

WHAT CINEGRAPH IS

An editor that ships a SKILL.md rewrite after every run.

INPUT

A human edit you want to imitate.

A short video the human cut by hand. The agent never sees their reasoning — just the result.

OUTPUT

An agent that gets closer each run.

A wiki of SKILL.md files — covering cut detection, b-roll, pacing, transitions, on-screen text — rewritten by Cognee, not by hand.

THE LOOP

ingest → attempt → critique → self-improve → re-attempt

Every step is a single CLI command.

THE STAKE

Same agent. Smarter wiki.

No new model. No new prompt. The improvement lives in the markdown.

To make this concrete: input is one 10-second human-edited video. Output is five SKILL.md files that started as generic editing rules and end up specialized to this kind of content. The agent's brain is the wiki — that's the artifact that improves.

What Cognee does for us.

The permanent memory. The wiki that compounds.

THE JOB
  • holds every SKILL.md as a graph node
  • turns each critique into a SkillRunEntry
  • proposes a rewrite, then commits it on apply
  • keeps every past proposal — the wiki compounds across runs
THE LOOP, IN CODE
await cognee.remember(
    "./my_skills", content_type="skills")

await cognee.remember(
    SkillRunEntry(skill_id=..., score=..., feedback=...),
    skill_improvement={"apply": False})

await improve_skill(..., apply=True)

The propose-then-apply cycle is what "self-improvement" means here — and Cognee gives it to us out of the box.

Cognee is the permanent layer. It holds every SKILL.md as a graph node, turns each critique into a SkillRunEntry, proposes a rewrite, and commits it when we call apply=True. The graph remembers every past proposal — that's what makes the wiki compounding rather than just stateful. The propose-then-apply cycle is exactly the self-improvement pattern the hackathon asked for, and we get it for free from Cognee's API.

— AND ON THE HOT SIDE —

What Redis does for us.

The session memory. The live nervous system of every run.

SESSION MEMORY

cognee.remember(..., session_id=) routes per-run scratchpad straight to Redis. Forgets between runs by design.

VECTOR SEARCH

RediSearch HNSW index over source-clip descriptions — sub-ms semantic retrieval of which clip to grab.

STREAMS

Every pipeline step (ingest · attempt · critique · self-improve · memory · lint) XADDs to llmwiki:events. Replayable audit log.

PUB/SUB

The viz subscribes to llmwiki:live — events animate the dashboard as the agent works. The live stream you saw in the demo.

one Redis instance · four jobs · drives the entire demo

Redis does four jobs at once. Session memory holds the per-run scratchpad and forgets between runs by design. RediSearch gives the agent semantic clip lookup. Streams record every pipeline event — ingest, attempt, critique, self-improve, memory writes, lint — as an XADD, so the whole timeline is replayable. And pub/sub drives the live dashboard you saw during the demo. One Redis instance, four distinct features, all wired through cognee's session_id parameter.
